# cloud-learn-exp03（实践教程3：系统监控与AOP切面）

本实践的目标不是“写出更多业务功能”，而是让你亲手体会：当系统需要日志、性能监控、事务等横切能力时，如何用 Spring AOP 把这些能力从业务代码中剥离出来，做到“业务专注业务，系统能力由切面统一增强”。

> 建议按步骤完成：先保证没有 AOP 的业务能跑通，再逐步引入 3 个切面，观察输出变化。

---

## 学习目标

- 用 Spring Framework搭建一个最小可运行的 IoC 容器项目
- 写一组“线上课程相关”的业务类（DAO + Service），用它们承载横切增强
- 通过 3 个切面分别实现：
  - 日志记录（理解 5 种通知里最常用的 4 种）
  - 性能监控（环绕通知 + 统计报告）
  - 事务模拟（环绕通知 + 异常触发回滚路径）
- 通过运行 `Main` 对比 AOP 生效前后的行为差异，输出就是你的“观察结果”

## 第一阶段：使用“没有 AOP 也能跑”的业务作为基线

目标：在不引入任何切面的情况下，让 `Main` 能创建 Spring 容器、拿到 `Service` Bean、完成查询/更新。

你需要完成的事情：

1) 创建课程实体 `Course`  
- 至少包含：`id`、`title`、`studentCount`（方便观察人数变化）
- 给它一个可读的 `toString()`，后续日志会打印返回值

2) 设计 DAO 接口与实现  
- DAO 负责“像数据库一样”保存/查询课程
- 实现类建议用线程安全容器（如 `ConcurrentHashMap`）作为内存库
- 在 DAO 初始化时预置几条课程数据（这样 `findAll` 能返回内容）
  - 建议使用生命周期回调（如 `@PostConstruct`）来模拟“系统启动加载数据”

3) 设计 Service 接口与实现  
- Service 只写业务：查课程、搜索课程、增加学习人数等
- 这里不要写日志、不要写耗时统计、不要写事务语义（这些都留给 AOP）

4) 写一个能跑的 `Main`  
- 用 `AnnotationConfigApplicationContext` 启动容器
- 从容器里拿到 `CourseService` 并调用几次方法
- 先确认：能看到初始化数据、能查询到课程、能触发一次“课程不存在”的异常

使用以下命令：

```bash
mvn -q -DskipTests compile exec:java -Dexec.args="baseline"
```

检查点：

- 请延伸思考一下，为什么 Service 里不应该直接 `new CourseDaoImpl()`，而是交给 Spring 注入？
- 你能在控制台看到：容器启动 + 业务方法调用的基本输出

## 第二阶段：开启 AOP，让 Spring 具备“织入能力”

目标：让 Spring 能为目标对象创建代理，从而在调用前后“插入”你的横切逻辑。

你需要做的事情：

1) 在 `pom.xml` 增加 AOP 相关依赖  
- Spring AOP（提供 AOP 能力）
- AspectJ Weaver（提供切点表达式/织入支持）

2) 在配置类开启 AOP 自动代理  
在 `config/AppConfig` 中完成两件事：

- 组件扫描：确保 `service/dao/aspect` 都能被 Spring 扫描为 Bean
- 开启 AOP：让 Spring 在运行时用代理包装你的目标 Bean

检查点：

- 请延伸思考一下，为什么“没有开启 AOP”时，写了切面类也不会生效
- 请理解一下：AOP 生效的本质是“容器返回的是代理对象，而不是原对象”

## 第三阶段：日志切面，让业务方法“自动被记录”

目标：对 Service 层方法统一记录调用信息，包括：

- 调用前：方法名、参数
- 调用后：无论成功失败都记录“结束”
- 成功返回：打印返回值
- 抛出异常：打印异常信息

你需要做的事情：

1) 定义一个切入点（Pointcut）  
- 选择一个清晰的规则：例如“匹配 service 包下所有方法”
- 让日志逻辑只关注这一类方法，避免把 DAO 也打满日志

2) 编写通知（Advice）  
建议按以下顺序添加并观察输出变化：

- 前置通知@Before：先看到“调用开始”
- 后置通知@After：总能看到“调用结束”
- 返回通知@AfterReturning：只有成功时打印返回结果
- 异常通知@AfterThrowing：只有抛异常时打印错误信息

3) 运行 `Main` 的日志模式观察  
- 正常查询：应该出现“前置 -> 返回 -> 后置”
- 异常查询：应该出现“前置 -> 异常 -> 后置”

使用以下命令：

```bash
mvn -q -DskipTests compile exec:java -Dexec.args="log"
```

检查点：

- 请延伸思考一下，为什么 `@After` 在异常时也会执行
- 请延伸思考一下，为什么 `@AfterReturning` 不会在异常时执行

## 第四阶段：性能监控切面，掌控环绕通知

目标：统一统计 Service/DAO 方法的执行耗时与调用次数，并在结束时输出报告。

你需要做的事情：

1) 选择“监控范围”  
- 建议同时监控 `service` 和 `dao`（便于观察调用链）
- 切点可以用“或”组合规则

2) 用环绕通知实现“计时 + 执行 + 统计”  
环绕通知要完成 3 件事：

- 执行前记录开始时间
- 调用 `proceed()` 让目标方法继续执行
- 成功/失败都要计算耗时并更新统计

3) 设计一个“专门用于制造耗时差异”的服务  
- 准备快/中/慢三个方法（用 sleep 模拟）
- 再准备一个批量方法（循环调用）

4) 打印统计报告  
- 报告建议包含：方法、调用次数、平均耗时、最大耗时、成功率

使用以下命令：

```bash
mvn -q -DskipTests compile exec:java -Dexec.args="perf"
```

检查点：

- 请延伸思考一下，为什么“只有环绕通知”才能做到完整的计时（前后包裹目标执行）
- 请延伸思考一下，为什么统计容器要考虑并发安全（即使你现在是单线程演示）

## 第五阶段：事务模拟切面，理解异常顺序

目标：用环绕通知模拟事务边界，观察成功时“提交”、异常时“回滚”的控制流。

你需要做的事情：

1) 选择“事务方法”的判定规则  
- 常见规则：方法名以 `save/update/delete/create` 开头
- 你也可以把本项目的 `add/increment/enroll` 等写操作纳入规则，方便演示

2) 编写事务环绕逻辑（模拟即可）  
你需要在控制台输出类似流程：

- 开启连接、关闭自动提交（开始）
- 执行业务方法
- 成功：提交
- 失败：回滚并继续把异常抛出去（让上层能感知）
- 最终：关闭连接

3) 准备一个“必定失败”的业务路径  
- 例如选课流程中，最后一步模拟支付服务不可用，直接抛异常
- 观察输出：事务切面进入回滚路径、日志切面触发异常通知、性能切面打印失败耗时

使用以下命令：

```bash
mvn -q -DskipTests compile exec:java -Dexec.args="tx"
```

检查点：

- 请延伸思考一下：为什么异常需要继续抛出？
- 请延伸思考一下，多个切面叠加时，输出顺序为什么会是“像洋葱一样一层层进/出”

## 运行与观察建议

本项目入口支持 4 种模式（通过启动参数选择）：

- `log`：聚焦日志通知类型差异
- `perf`：聚焦环绕通知计时与统计
- `tx`：聚焦异常路径与回滚
- `all`：串联观察多个切面叠加时的输出顺序

示例：

```bash
mvn -q -DskipTests compile exec:java -Dexec.args="tx"
```

观察要点：

- 同一个业务方法调用，会同时被多个切面增强
- 同一个异常，会被：性能切面记录失败、日志切面记录异常通知、事务切面触发回滚

## 常见问题（排错路径）

1) Maven 提示找不到 POM  
- 你在错误目录执行了命令；请在 `cloud-learn-exp03` 目录运行，或用 `-f` 指定 pom

2) 切面不生效（没有任何 AOP 输出）  
- 检查是否开启了 AOP 自动代理
- 检查切面类是否被扫描为 Bean（包名是否在扫描范围内）
- 检查切点表达式是否匹配到了你调用的方法

3) 输出顺序看不懂  
- 先只开一个切面观察，再把多个切面叠加
- 从“环绕通知进入 -> proceed -> 环绕通知退出”的视角理解嵌套关系

## 课后加深

- 权限切面：对某些 Service 方法做“登录校验”（不通过则直接抛异常）
- 缓存切面：对查询方法做简单缓存（命中则不再调用 DAO）
- 重试切面：对不稳定方法失败时自动重试 N 次（使用环绕通知实现）
